Language-Specific Security Patterns

Each programming language has unique security characteristics, common vulnerability patterns, and best practices. Mend's SAST engine includes language-specific rules that understand the nuances of each ecosystem.

Supported Languages

☕

Java / Kotlin

Deserialization, SQL injection via JDBC/JPA, XXE in XML parsers, Spring Security misconfigurations.

🐍

Python

Pickle deserialization, Jinja2 template injection, SQL injection in Django/Flask ORM, SSRF via requests library.

📚

JavaScript / TypeScript

Prototype pollution, XSS in React/Angular, eval() injection, insecure regex, path traversal in Express.

🔧

C# / .NET

SQL injection via raw ADO.NET, insecure deserialization (BinaryFormatter), XSS in Razor views, SSRF.

Java Security Deep Dive

Java applications face specific security challenges due to the language's rich serialization support and extensive framework ecosystem:

Common Java Vulnerabilities

// VULNERABLE: SQL Injection via string concatenation
String query = "SELECT * FROM users WHERE id = " + userId;
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(query);

// SECURE: Parameterized query
String query = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setInt(1, userId);
ResultSet rs = pstmt.executeQuery();

// VULNERABLE: Unsafe deserialization
ObjectInputStream ois = new ObjectInputStream(untrustedInput);
Object obj = ois.readObject();  // Can execute arbitrary code

// SECURE: Use allowlist-based deserialization
ObjectInputFilter filter = ObjectInputFilter.Config.createFilter(
    "com.myapp.model.*;!*"
);
ois.setObjectInputFilter(filter);

JavaScript/TypeScript Security

JavaScript's dynamic nature creates unique security challenges:

// VULNERABLE: Prototype pollution
function merge(target, source) {
    for (let key in source) {
        target[key] = source[key]; // Can modify Object.prototype
    }
}
// Attack: merge({}, JSON.parse('{"__proto__":{"isAdmin":true}}'))

// SECURE: Check for prototype pollution
function safeMerge(target, source) {
    for (let key in source) {
        if (key === '__proto__' || key === 'constructor' || key === 'prototype') continue;
        if (!source.hasOwnProperty(key)) continue;
        target[key] = source[key];
    }
}

// VULNERABLE: XSS in React (rare but possible)
function UserProfile({ bio }) {
    return <div dangerouslySetInnerHTML={{__html: bio}} />;
}

// SECURE: Use DOMPurify for sanitization
import DOMPurify from 'dompurify';
function UserProfile({ bio }) {
    return <div dangerouslySetInnerHTML={{__html: DOMPurify.sanitize(bio)}} />;
}

Python Security Considerations

Python's flexibility and dynamic typing create specific security concerns:

# VULNERABLE: Pickle deserialization (arbitrary code execution)
import pickle
data = pickle.loads(untrusted_bytes)  # Never do this!

# SECURE: Use JSON or MessagePack for untrusted data
import json
data = json.loads(untrusted_string)

# VULNERABLE: SQL injection in Django raw queries
User.objects.raw("SELECT * FROM users WHERE name = '%s'" % name)

# SECURE: Use parameterized queries
User.objects.raw("SELECT * FROM users WHERE name = %s", [name])

# VULNERABLE: SSTI (Server-Side Template Injection)
from jinja2 import Template
Template(user_input).render()  # User controls the template

# SECURE: Never let users control templates
from jinja2 import Environment, select_autoescape
env = Environment(autoescape=select_autoescape(['html', 'xml']))
template = env.get_template('fixed_template.html')
template.render(user_data=user_input)

Configure Mend SAST to use language-specific rule sets for each repository. A Java Spring Boot API needs different rules than a React frontend. Tailored rule sets reduce false positives and catch language-specific issues that generic rules miss.
