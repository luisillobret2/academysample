False Positive Management

🕒 30 min 📚 4 sections ⭐ 150 XP Level 2 - Intermediate

🎯 Learning Objectives

In this module you will cover:

  • Identify and classify false positives in SAST results
  • Apply suppression and auto-suppression workflows in Mend SAST
  • Tune rules and improve the signal-to-noise ratio of SAST findings

Identifying False Positives

A false positive is a SAST finding that reports a vulnerability where none actually exists. Common causes include over-generalized rules, missing context about sanitization, framework-specific controls that the engine does not understand, and dead code that is never executed. False positives waste time, reduce trust in security tools, and contribute to alert fatigue.

Recognizing a false positive requires both security knowledge and code context. Ask whether the reported source is actually attacker-controlled, whether the sink is actually dangerous, and whether a sanitizer or validation step exists between them. A SQL query that concatenates a constant string is not injection, and user input passed through a prepared statement is not vulnerable even if the syntax looks similar to an unsafe pattern.

False Positive vs. Low Risk

A false positive is factually incorrect. A low-risk finding is real but not worth fixing immediately. It is important to distinguish the two. Suppressing a low-risk finding because it is inconvenient hides real risk, while suppressing a true false positive reduces noise. Document the rationale for every suppression.

Common False Positive Patterns

🔍

Hardcoded Constants

String concatenation that uses hardcoded values or constants is often flagged as injection but is not exploitable.

🔓

Built-in Sanitizers

Frameworks such as Razor, React, and Django auto-escape output. SAST may not always recognize these built-in controls.

💀

Dead Code

Code that is unreachable in production, such as test utilities or commented-out debug paths, can trigger false findings.

📜

Custom Validators

Custom input validation or allowlists may not be modeled by default rules and can be misclassified as missing.

Suppression Workflows

Mend SAST provides several ways to suppress findings that are confirmed as false positives. Inline suppression comments are placed directly in source code and are visible to developers reviewing the file. Suppression from the platform UI lets security teams triage findings without modifying code. The choice depends on who owns the decision and whether the suppression needs to be version-controlled.

Every suppression should include a rationale and an expiration date if possible. A suppression without context is a future mystery. When the underlying rule improves or the code changes, a dated suppression forces re-evaluation. Mend captures suppression metadata in the platform so auditors can review why a finding was dismissed.

Inline Suppression Examples

// Java: suppress a false positive SQL injection finding where the input is hardcoded
// mend-sast-suppress SQLInjection reason="Query uses hardcoded status values only" until="2026-12-31"
String query = "SELECT * FROM orders WHERE status = '" + status + "'"; // status is from enum

// Python: suppress a false positive path traversal when a fixed base path is enforced
# mend-sast-suppress PathTraversal reason="Base path is restricted to /app/data via os.path.join validation" until="2026-12-31"
with open(os.path.join(BASE_DIR, filename)) as f:
    data = f.read()

// JavaScript: suppress a false positive XSS when output is rendered through a trusted framework
// mend-sast-suppress XSS reason="Output is rendered by a framework that auto-escapes HTML" until="2026-12-31"
return <div>{userBio}</div>;

Auto-Suppression Rules

Auto-suppression rules reduce repetitive triage by suppressing a class of findings that match a known safe pattern. For example, if every route in an API uses a specific authentication middleware that SAST does not model, an auto-suppression rule can dismiss the corresponding missing-auth findings. Rules can match by file path, rule ID, function name, or code pattern.

Auto-suppression must be governed carefully. A rule that is too broad can hide real vulnerabilities. Review auto-suppression rules quarterly, limit them to specific repositories or applications, and require security approval for new rules. Good rules are narrow, well-documented, and tied to a specific control that compensates for the reported issue.

💡 Triage Best Practice

Never auto-suppress a finding based only on a file path. Combine path, rule ID, and a code pattern or control evidence so that the suppression only applies when the compensating control is present. This prevents real vulnerabilities from being dismissed when the file changes.

Tuning Rules & Feedback Loops

Rule tuning is the long-term solution to false positives. If a rule repeatedly reports safe patterns, adjust its sensitivity, add sanitization sinks, or scope it to files where the issue is actually relevant. Mend SAST lets teams customize rule sets, disable noisy rules, and calibrate taint analysis for their specific codebases.

Feedback loops between security teams and the Mend team improve the product for everyone. When a false positive is caused by a gap in the engine, report it with the rule ID, a minimal code sample, and the expected behavior. Mend can then update the rule or the taint model so the next scan is more accurate. Regularly review the top noisy rules and prioritize tuning the ones with the highest false positive rates.

Signal-to-Noise Improvement Plan

  1. Measure the false positive rate by rule and by application over the last 30 days.
  2. Suppress individual confirmed false positives with documented rationale.
  3. Create narrow auto-suppression rules for recurring safe patterns.
  4. Disable or tune rules that consistently produce more noise than value.
  5. Report engine-level false positives to Mend for rule improvement.
  6. Re-measure monthly and iterate.
