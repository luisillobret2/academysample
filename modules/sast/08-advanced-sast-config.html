Custom Rule Authoring

Custom rules let security teams encode organization-specific threats that are not covered by the default rule set. For example, a company might have a custom authentication wrapper, an internal cryptography library, or a banned API that default rules do not know about. Custom rules can detect these patterns and enforce internal standards consistently.

When writing a custom rule, define the source, the sink, and any sanitizers or validators in between. A source is where untrusted data enters the application, such as an HTTP request parameter or a database field. A sink is where the data becomes dangerous, such as an HTML renderer or a command executor. Sanitizers are functions that neutralize the data. A well-written rule models the real data flow rather than just matching a string pattern.

Rule Anatomy

Every custom rule needs a unique ID, a clear description, a severity, a set of patterns, and a remediation message. Good rules are specific enough to avoid false positives and general enough to catch the variants that developers will create. Test custom rules against a corpus of real code before deploying them to production scans.

Example Custom Rule Pattern

rule: internal_banned_crypto
id: CSP-1001
severity: high
message: Use of deprecated internal crypto helper detected
sources:
  - any
sinks:
  - pattern: "LegacyCrypto.encrypt(*)"
  - pattern: "LegacyCrypto.decrypt(*)"
sanitizers: []
remediation: Replace with company.crypto.SecureCipher from the approved library.

Taint Analysis Configuration

Taint analysis traces untrusted data from sources to sinks. Mend SAST models common sources, sinks, and sanitizers for each supported language, but every application has unique boundaries. Advanced configuration lets you extend the taint model with custom sources, sinks, and sanitizers so the engine understands your frameworks and libraries.

Data flow tuning controls how the engine follows data through the program. You can increase precision for security-critical code paths, reduce analysis depth for generated code, or mark certain methods as opaque barriers that taint cannot pass through. These settings trade off between scan time, memory usage, and result accuracy.

🖍

Custom Sources

Mark internal APIs, message queues, or configuration stores as taint sources when they receive untrusted data.

🕳

Custom Sinks

Define organization-specific dangerous operations such as custom loggers, RPC clients, or legacy database access.

🔒

Sanitizers

Teach the engine that an internal validation function neutralizes input for a particular vulnerability class.

🔬

Flow Sensitivity

Adjust how deeply the engine tracks data flow across functions, conditionals, and object fields.

Exclusions & Scan Scope Optimization

Not every file needs to be scanned. Generated code, third-party dependencies, test data, and build artifacts can be excluded from analysis to reduce scan time and false positives. Exclusions should be specific: exclude a directory or a glob pattern rather than entire modules without justification. Document why each exclusion exists and review it periodically.

Scan scope optimization also includes choosing which rules to run. A full rule set is appropriate for a security audit, but a CI scan for fast feedback may use only high-confidence rules for critical and high severity issues. Mend SAST lets you define multiple profiles for different contexts, such as "PR fast scan", "nightly deep scan", and "release audit".

Exclusion Configuration

exclude:
  - "**/node_modules/**"
  - "**/dist/**"
  - "**/generated/**"
  - "**/tests/fixtures/**"
  - "*.min.js"
  - "*.designer.cs"

rules_profile:
  name: pr-fast-scan
  severities: [critical, high]
  confidence: high
  include_rules: []
  exclude_rules: ["style", "informational"]

💡 Scope Tip

Audit your exclusions every quarter. Code that was generated or third-party when excluded may become first-party code later. Stale exclusions are a common cause of missed vulnerabilities and unexpected false positives.

Incremental Scanning & Performance Profiling

Incremental scanning analyzes only the parts of the codebase that changed since the last baseline. This is essential for monorepos and large codebases where full scans can take tens of minutes. Mend SAST incremental mode compares the current commit to the previous analysis result, reuses cached summaries, and re-scans only modified functions and their dependencies.

Performance profiling identifies the slowest stages of a scan. Common bottlenecks include parsing large generated files, deep interprocedural analysis across many layers, and scanning dependencies that should be excluded. Use the CLI's profiling flags to produce a timing breakdown by stage. Use that data to tune exclusions, reduce analysis depth, or split the scan into smaller profiles.
