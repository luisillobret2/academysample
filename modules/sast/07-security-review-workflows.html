Pull Request Integration

Pull request integration is the most effective way to catch vulnerabilities before they reach the main branch. Mend SAST scans changed files when a PR is opened or updated, and reports findings directly in the Git provider's interface. This keeps feedback in the developer's natural workflow and avoids the need to switch between security tools and code review tools.

PR scans are faster and more focused than full baseline scans. By analyzing only the diff, Mend reduces scan time and surfaces findings that are relevant to the current change. Developers can see the vulnerable line, the rule that triggered, and remediation guidance without leaving the PR. This immediacy is the foundation of a shift-left security program.

📈

Diff-Only Scanning

Scan only the files and functions changed in the PR, reducing feedback time from minutes to seconds.

💬

Inline Code Review Comments

Inline comments turn SAST findings into actionable code review feedback. A good comment explains why the code is risky, shows what an attack might look like, and suggests a concrete remediation. Developers are more likely to fix issues when the feedback is clear, specific, and respectful rather than generic or alarmist.

Mend SAST can include remediation snippets directly in the comment, suggesting a safer pattern for the same language and framework. For example, a SQL injection finding might include a prepared statement example, while a cross-site scripting finding might show the correct output encoding function. These snippets reduce the cognitive load on developers and speed up remediation.

Comment Quality

High-quality comments include: the vulnerability name, the affected line, the attack scenario, the safe pattern, and a link to documentation. Avoid comments that only say "vulnerability found" or assign blame. The goal is to teach the developer, not to shame them.

Example Inline Comment

File: src/api/orders.js, line 42
Finding: SQL Injection (CWE-89)
Severity: High

The userId parameter is concatenated directly into a SQL query. An attacker
could provide a value such as "1 OR 1=1" to access other users' orders.

Suggested fix: use a parameterized query.

const query = "SELECT * FROM orders WHERE user_id = ?";
const rows = await db.query(query, [userId]);

Security Gates & Approval Workflows

Security gates enforce minimum standards before code can be merged or deployed. A gate can block PRs that introduce critical or high-severity vulnerabilities, require approval for new medium-severity issues, or allow merge only after all findings are resolved or approved. Gates must be calibrated to the team's maturity: too strict and developers find workarounds, too lenient and vulnerabilities slip through.

Approval workflows let security champions review exceptions. When a developer believes a finding is a false positive or accepts the risk, they can request an approval from a security engineer. The approver can see the finding, the code, the developer's rationale, and the security impact. Approved exceptions are logged for audit and revisited periodically.

Gate LevelConditionTypical Action
BlockCritical or high severity findingPrevent merge until fixed or approved
WarnMedium severity findingAllow merge with security review required
InformLow severity or informationalComment only, no block
ExceptionApproved risk acceptanceMerge allowed with documented exception

Shift-Left Security Culture

Shift-left security means addressing security as early as possible in the development lifecycle. SAST in the PR workflow is a core shift-left practice because it gives feedback while code is still being written. But tools alone do not create culture. Developers need training, clear policies, fast feedback, and trust that security findings are relevant and helpful.

Build feedback loops between security and engineering teams. Regularly review the most common findings, the slowest-remediated categories, and the top sources of false positives. Use this data to update training, tune rules, and improve documentation. When developers see that their input leads to better tools, they are more likely to engage with security findings.

💡 Workflow Tip

Keep PR scan time under five minutes. If scans take longer, developers will start bypassing them. Use diff-only scanning, incremental analysis, and rule set optimization to stay within the feedback window that developers expect.
