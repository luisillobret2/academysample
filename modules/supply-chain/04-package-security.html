Package Manager Security

🕐 35 min 📚 4 sections ⭐ 175 XP Level 2 - Intermediate

Package Manager Attack Surfaces

Package managers (npm, PyPI, Maven, NuGet, RubyGems) are the primary distribution mechanism for open-source software. Each ecosystem has unique security characteristics and attack vectors that organizations must understand.

Common Attack Patterns

  • Typosquatting — Registering packages with names similar to popular ones (e.g., lodahs instead of lodash)
  • Dependency Confusion — Publishing a higher-version public package with the same name as a private internal package
  • Account Takeover — Compromising a maintainer's account to publish malicious updates
  • Install Scripts — Malicious postinstall, preinstall, or setup.py scripts that execute during installation
  • Star Jacking — Linking a malicious package's repository URL to a legitimate popular project

Ecosystem-Specific Security

npm (JavaScript/Node.js)

# Use lockfiles to pin exact versions
npm ci  # Clean install from lockfile (never npm install in CI)

# Audit for known vulnerabilities
npm audit

# Enable package-lock.json enforcement
# .npmrc
package-lock=true
save-exact=true

# Scope private packages to prevent dependency confusion
@mycompany:registry=https://npm.mycompany.com

Python (pip/Poetry)

# Pin versions with hashes for integrity verification
pip install --require-hashes -r requirements.txt

# requirements.txt with hashes
requests==2.31.0 \
    --hash=sha256:58cd2187c01e70e6e26505bca751777aa9f2ee0b7f4300988b709f44e013003eb

# Use a private index for internal packages
[global]
extra-index-url = https://pypi.mycompany.com/simple/

Maven (Java)

<!-- Enforce dependency verification -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-enforcer-plugin</artifactId>
    <configuration>
        <rules>
            <requireReleaseDeps>
                <message>No SNAPSHOT dependencies allowed in release builds</message>
            </requireReleaseDeps>
            <banDuplicatePomDependencyVersions/>
        </rules>
    </configuration>
</plugin>

Registry Security Controls

Organizations should implement controls at the registry level to prevent malicious packages from entering their build pipelines:

  • Private Registry Proxies — Use Artifactory, Nexus, or similar tools to proxy and cache public packages with scanning
  • Allowlist Policies — Only permit pre-approved packages in sensitive projects
  • Namespace Reservation — Register your organization's namespace on public registries to prevent dependency confusion
  • Integrity Verification — Verify package checksums and signatures before installation

Mend Supply Chain Security Features

Mend provides several capabilities specifically designed to protect against supply chain attacks through package managers:

🛡

Malicious Package Detection

Behavioral analysis identifies packages with suspicious install scripts, data exfiltration, or reverse shells.

🔍

Reputation Scoring

Evaluates package age, maintainer history, download patterns, and code changes to assess trustworthiness.

🔔

Real-Time Alerts

Immediate notification when a dependency you use is flagged as malicious or compromised.

📄

Policy Enforcement

Block builds that include packages below minimum age, reputation score, or without verified provenance.

💡 Best Practice

Configure Mend to block any new dependency published less than 7 days ago. Most supply chain attacks are detected within the first week of publication.
