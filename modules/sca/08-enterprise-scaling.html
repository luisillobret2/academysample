Multi-Organization Management

Large enterprises rarely operate as a single team. Mend SCA supports a hierarchical structure of organizations, applications, and projects that maps to business units, product lines, and repositories. This hierarchy lets security teams delegate ownership while maintaining centralized visibility and governance.

Each organization can inherit base policies from a parent entity or define its own. Application owners see only their own scan results, while CISO-level dashboards aggregate risk across every organization. Role-based access control ensures that developers, security engineers, and executives each see the data they need without exposing sensitive information from unrelated teams.

🏢

Organization

Top-level boundary for business units or subsidiaries. Contains applications, users, policies, and integrations.

💻

Application

Logical grouping of related projects, such as a microservice product or a frontend/backend pair.

📚

Project

Maps to a single repository or scan target. The level where most scan results are reported and remediated.

🔐

Role-Based Access

Roles such as Admin, Application Manager, and Contributor control who can view, modify, or approve findings.

Scanning Thousands of Repositories

At enterprise scale, scanning every repository on every commit can overwhelm CI infrastructure and produce more data than teams can act on. A scalable SCA program uses a mix of strategies: selective scanning, incremental scanning, baseline scans, and parallel execution.

Selective scanning focuses on repositories that contain build files or dependency manifests. Incremental scanning compares the current commit to the previous baseline and only rescans changed dependencies. Baseline scans run nightly or weekly to catch newly disclosed vulnerabilities and license changes that do not trigger CI commits. Parallel execution distributes scans across build agents to reduce queue time.

Deduplication Strategy

Shared libraries and common dependencies appear in hundreds of repositories. Deduplication groups findings by component, CVE, and remediation path so teams see one actionable issue instead of hundreds of identical alerts. Mend's "effective vulnerability" view collapses duplicate instances across the enterprise.

Scaling Scan Configuration

# Mend CLI: scan only changed dependencies in a monorepo
mend sca scan --diff-only --baseline baseline-report.json

# Mend CLI: scan multiple repositories in parallel with a manifest file
mend sca scan --batch repos.txt --workers 8 --output-format json

# CI/CD: run nightly baseline for newly disclosed CVEs
mend sca scan --baseline-only --schedule cron="0 2 * * *"

Centralized Policy & Reporting

Centralized policy management ensures that every team follows the same minimum security bar without requiring manual configuration in every repository. Policies are defined once at the organization level and inherited by applications and projects. Teams can add stricter rules, but they cannot silently weaken enterprise requirements.

Cross-team reporting aggregates findings by severity, component, remediation status, and owner. Executive dashboards show enterprise-wide trends, while operational dashboards help AppSec teams track which repositories are lagging. The Mend API 3.0 makes it possible to pull these metrics into existing business intelligence tools, data lakes, or governance dashboards.

🛠

Policy Inheritance

Define policies at the enterprise level and let applications inherit them. Local overrides are visible and auditable.

📊

Metrics APIs

Export vulnerability counts, MTTR, policy violations, and license findings to BI tools via the Mend API.

🔄

Automated Remediation

API-driven workflows can open pull requests, create tickets, and notify owners at scale without manual intervention.

🐝

Large Monorepos

Monorepo optimization scans only affected workspaces, caches dependency resolution, and distributes analysis across agents.

Performance Tuning for Large Monorepos

Monorepos with millions of lines of code and thousands of dependencies pose unique performance challenges. The key tuning levers are scope control, dependency resolution caching, and scan frequency. Scope control limits analysis to the packages or services affected by a change. Dependency resolution caching reuses resolved graphs when lock files have not changed. Scan frequency optimization runs full scans less often and incremental scans more often.

Profile scan duration by stage to identify bottlenecks. If dependency resolution dominates, optimize caching and manifest parsing. If reporting dominates, reduce the number of exported formats or aggregate data server-side. Mend's CLI includes timing and profiling options to help teams find the right balance.

💡 Scale Tip

Start with a representative set of repositories and measure scan time, CI queue time, and findings volume before rolling out to the entire enterprise. Use these measurements to set realistic service-level objectives and to tune scan scope before broad deployment.
