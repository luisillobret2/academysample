Understanding Container Image Layers

Container images are built in layers, each adding files and dependencies on top of a base image. Vulnerabilities can exist at any layer — from the OS packages in the base image to the application dependencies installed during the build process.

▶

Where Vulnerabilities Hide

  • Base Image OS Packages — Ubuntu, Alpine, Debian base images contain OS-level packages (openssl, glibc, curl) that may have known CVEs
  • Language Runtime — The Node.js, Python, Java, or Go runtime installed in the image may have vulnerabilities
  • Application Dependencies — npm, pip, Maven packages installed during docker build
  • Custom Binaries — Compiled binaries copied into the image may link against vulnerable libraries
  • Configuration Files — Exposed ports, hardcoded credentials, overly permissive file permissions

💡 Base Image Selection

Start with minimal base images (Alpine, distroless, or scratch) to reduce your attack surface. A typical ubuntu:latest image contains 100+ OS packages; alpine:latest contains fewer than 20.

Scanning with Mend

Mend scans container images at multiple stages of the development lifecycle:

💻

Local Development

Scan images on your workstation before pushing to a registry. Catch vulnerabilities during development.

⚙️

CI/CD Pipeline

Automatically scan images during build. Block deployment if vulnerabilities exceed policy thresholds.

📦

Registry Scanning

Continuously monitor images in Docker Hub, ECR, ACR, GCR, and other registries for new vulnerabilities.

☁️

Runtime Monitoring

Monitor running containers in Kubernetes clusters for newly discovered vulnerabilities.

CLI Scanning Commands

# Scan a local image
mend image scan myapp:latest

# Scan an image from a registry
mend image scan registry.example.com/myapp:v2.1

# Scan a Dockerfile (before building)
mend image scan --dockerfile ./Dockerfile

# Scan with policy enforcement
mend image scan myapp:latest --policy production --fail-on critical

# Output results in JSON
mend image scan myapp:latest --format json --output results.json

Remediation Strategies

When vulnerabilities are found in container images, Mend recommends specific remediation actions:

Base Image Upgrade

# BEFORE: Vulnerable base image
FROM node:18.15.0-bullseye

# AFTER: Updated base image with patched OS packages
FROM node:18.20.0-bullseye

# BEST: Minimal image with smallest attack surface
FROM node:18.20.0-alpine

Multi-Stage Builds

# Stage 1: Build (includes dev dependencies)
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production (minimal runtime only)
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/server.js"]

💡 Image Size Matters

Multi-stage builds reduce image size by 60-80% by excluding build tools, source code, and dev dependencies. Smaller images have fewer vulnerabilities and deploy faster.
