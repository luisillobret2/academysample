Runtime Protection & Monitoring

🕐 40 min 📚 4 sections ⭐ 225 XP Level 3 - Advanced

Beyond Build-Time Security

Scanning containers at build time is essential but not sufficient. Runtime protection detects and prevents attacks against running containers, covering threats that can't be identified by static analysis alone.

The Security Lifecycle

🔧

Build

Scan images during CI/CD. Fix known vulnerabilities and enforce base image policies before deployment.

📦

Ship

Registry scanning, image signing, and admission control ensure only verified images are deployed.

💻

Run

Runtime monitoring detects anomalous behavior, unauthorized access, and zero-day exploits in production.

📊

Monitor

Continuous visibility into container behavior, network connections, file access, and process execution.

Runtime Threat Detection

Runtime security tools monitor container behavior and alert on anomalies:

  • Process Monitoring — Detect unexpected processes (e.g., a web server container running bash or curl)
  • File System Integrity — Alert when containers modify files outside expected paths
  • Network Anomalies — Identify unexpected outbound connections (data exfiltration, reverse shells)
  • Privilege Escalation — Detect containers attempting to gain elevated privileges
  • Cryptomining — Identify containers using excessive CPU for cryptocurrency mining

Falco Rules Example

# Detect shell execution in a container
- rule: Shell Spawned in Container
  desc: Detect shell execution in a running container
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh, ksh, csh) and
    not proc.pname in (expected_shell_parents)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
     image=%container.image.repository
     process=%proc.name parent=%proc.pname)
  priority: WARNING

# Detect outbound connections to suspicious ports
- rule: Unexpected Outbound Connection
  desc: Container making outbound connection to non-standard port
  condition: >
    outbound and container and
    fd.sport not in (80, 443, 53, 8080, 8443)
  output: >
    Unexpected outbound connection from container
    (container=%container.name image=%container.image.repository
     connection=%fd.name)
  priority: NOTICE

Network Policies

Kubernetes Network Policies restrict pod-to-pod communication, implementing zero-trust networking at the container level:

# Default deny all ingress traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Ingress

# Allow specific communication patterns
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - port: 8080
          protocol: TCP

Security Observability

Effective runtime protection requires comprehensive observability:

  • Centralized Logging — Aggregate container logs to a SIEM for correlation and analysis
  • Metrics & Alerting — Monitor resource usage patterns that may indicate compromise
  • Distributed Tracing — Track request flows across microservices to identify lateral movement
  • Audit Logs — Kubernetes API audit logs track all cluster operations for forensic analysis

💡 Defense in Depth

Runtime protection is most effective when combined with strong build-time scanning. Mend's vulnerability data enriches runtime alerts by identifying which running containers have known vulnerabilities, prioritizing response efforts.
