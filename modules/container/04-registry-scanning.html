Registry Scanning & Image Lifecycle

🕐 35 min 📚 4 sections ⭐ 200 XP Level 2 - Intermediate

Container Registry Security

Container registries are the distribution points for all container images in your organization. Securing registries is critical because a compromised registry can distribute malicious images across your entire infrastructure.

Registry Types & Security Features

📦

Docker Hub

Public registry with Docker Official Images, Verified Publishers, and automated vulnerability scanning for Pro/Team plans.

☁

Amazon ECR

AWS-native registry with IAM integration, image scanning on push, and cross-account sharing with fine-grained permissions.

🔐

Azure Container Registry

Geo-replicated registry with content trust, Microsoft Defender scanning, and Azure AD authentication.

🛡

Harbor

Open-source enterprise registry with built-in vulnerability scanning (Trivy), RBAC, and image signing.

Continuous Registry Scanning with Mend

Mend integrates with container registries to provide continuous scanning of all stored images, not just at push time:

# Configure Mend registry integration
mend container registry add \
    --type ecr \
    --region us-east-1 \
    --scan-schedule "0 */6 * * *"  # Scan every 6 hours

# Scan all images in a registry
mend container registry scan --registry myregistry.azurecr.io

# Set policies for registry images
mend container policy set \
    --max-critical 0 \
    --max-high 5 \
    --max-age-days 90 \
    --registry myregistry.azurecr.io

Why Continuous Scanning Matters

New vulnerabilities are discovered daily. An image that was clean at push time may have critical vulnerabilities days later. Continuous scanning ensures you're always aware of your current risk posture.

Image Lifecycle Management

Unmanaged image sprawl is a major security risk. Old, unpatched images accumulate in registries and may be deployed accidentally:

  • Retention Policies — Automatically delete images older than your retention period (e.g., 90 days for non-production)
  • Tag Immutability — Prevent tags from being overwritten to ensure reproducible deployments
  • Base Image Tracking — Track which production images are built on which base images for efficient patching
  • Vulnerability SLA — Define maximum acceptable time for an image to have critical/high vulnerabilities

Image Signing & Verification

Image signing ensures that only trusted, verified images are deployed to your environments:

# Sign an image with Cosign
cosign sign --key cosign.key myregistry.com/myapp:v1.0

# Verify an image signature
cosign verify --key cosign.pub myregistry.com/myapp:v1.0

# Kubernetes admission policy (Kyverno)
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  rules:
    - name: verify-cosign-signature
      match:
        resources:
          kinds: ["Pod"]
      verifyImages:
        - imageReferences: ["myregistry.com/*"]
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      ... your cosign public key ...
                      -----END PUBLIC KEY-----

💡 Best Practice

Combine Mend scanning with image signing: only sign images that pass your vulnerability threshold. Use admission controllers to enforce that only signed images can be deployed.
