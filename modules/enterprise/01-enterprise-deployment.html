Enterprise Deployment Architecture

🕐 45 min 📚 4 sections ⭐ 250 XP Level 3 - Expert

Deployment Models

Mend.io supports multiple deployment architectures to meet enterprise requirements around data residency, air-gapped networks, and scale. Understanding these models is essential for solution architects designing Mend deployments for large organizations.

▶

Enterprise Architecture Deep Dive

9:00
☁️

SaaS (Multi-Tenant)

Fully managed by Mend. Fastest time-to-value with automatic updates, zero infrastructure management. Data hosted in AWS US/EU regions.

🏛️

Dedicated SaaS

Single-tenant cloud instance with dedicated compute and storage. Meets compliance requirements for data isolation.

💻

Self-Hosted

Run Mend on your own infrastructure (on-premise or private cloud). Full control over data, networking, and updates.

🔒

Air-Gapped

For classified or highly regulated environments with no internet connectivity. Offline vulnerability database updates.

Multi-Organization Architecture

Large enterprises and MSPs often need to manage multiple organizations within a single Mend deployment:

Organization Hierarchy

Global Enterprise Account
├── Business Unit A (Organization)
│   ├── Team: Frontend
│   │   ├── Project: web-app
│   │   └── Project: mobile-app
│   └── Team: Backend
│       ├── Project: api-gateway
│       └── Project: payment-service
├── Business Unit B (Organization)
│   ├── Team: Platform
│   │   ├── Project: infrastructure
│   │   └── Project: shared-libs
│   └── Team: Data
│       └── Project: analytics-pipeline
└── Global Policies (inherited by all orgs)

    ├── License: Deny AGPL
    └── SBOM: Generate on release

Key Architecture Decisions

  • Organization Boundaries — Map to business units, subsidiaries, or customer tenants (for MSPs). Each org has independent users, projects, and policies
  • Policy Inheritance — Global policies cascade to all organizations. Orgs can add stricter policies but cannot relax global ones
  • SCM Integration — Each organization connects to its own GitHub/GitLab/Bitbucket/Azure DevOps instances
  • SSO Configuration — Support multiple identity providers — one per organization or a shared enterprise IdP
  • Data Segregation — Scan results, vulnerability data, and reports are isolated per organization

💡 Sizing Guidance

For self-hosted deployments, plan for 2 CPU cores and 4GB RAM per 500 repositories as a baseline. SCA and SAST scanning are the most compute-intensive operations. Container image scanning requires additional storage for layer caching.
