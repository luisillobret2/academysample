Security Architecture Review

🕒 40 min 📚 4 sections ⭐ 200 XP Level 2 - Intermediate

🎯 Learning Objectives

In this module you will cover:

  • How to conduct a structured customer security architecture review and assess current AppSec posture
  • How to map Mend SCA, SAST, Container, and Secrets capabilities to customer needs and gaps
  • How to produce a clear architecture review report with data flows, integration points, and remediation recommendations

The Architecture Review Process

A Security Architecture Review helps prospects and customers understand how Mend fits into their existing application security ecosystem. As a Technical SE, you act as a trusted advisor who translates customer requirements into a concrete Mend deployment design.

The review is not a sales pitch; it is a structured gap analysis. Your goal is to document the customer's current state, identify control gaps, and propose a future-state architecture that improves security coverage while minimizing disruption.

📋

Discover

Inventory current AppSec tools, languages, CI/CD platforms, SCMs, and compliance drivers.

🔍

Assess

Evaluate coverage, workflow friction, false-positive rates, and integration maturity.

💡

Design

Map Mend capabilities to gaps and propose a phased deployment architecture.

📝

Report

Document findings, diagrams, recommendations, and a roadmap for stakeholders.

💡 SE Tip

Always send a pre-work questionnaire before the review. Ask for the current tool list, tech stack, CI/CD pipeline overview, and top three AppSec pain points. This keeps the workshop focused and demonstrates professionalism.

Mapping Customer Needs to Mend Capabilities

Customers often express problems in business terms, such as "we have too many false positives" or "we need to secure our supply chain." Your job is to map these pain points to Mend's technical capabilities and show concrete value.

📦

Software Composition Analysis (SCA)

Need: Open-source risk, license compliance, SBOM generation.
Mend fit: Scan manifest files in repos and CI/CD; identify vulnerabilities, malicious packages, and license conflicts.

💻

Static Application Security Testing (SAST)

Need: Find custom-code vulnerabilities early.
Mend fit: IDE plugins and CI pipelines scan source code without requiring builds or test cases.

🐋

Container Security

Need: Secure images, registries, and Kubernetes deployments.
Mend fit: Scan container images and Dockerfiles for OS and application-layer vulnerabilities.

🔑

Secrets Detection

Need: Prevent leaked credentials in code and history.
Mend fit: Detect secrets in commits, pull requests, and repositories with remediation guidance.

When mapping capabilities, use a simple matrix: list each customer requirement, rate the current-state coverage, and show how Mend improves the rating. This format is easy for security and engineering stakeholders to understand.

Network and Security Considerations

Enterprise customers will ask about network paths, data residency, and encryption. Be prepared to answer:

  • Outbound traffic: Mend scanners typically need HTTPS (443) access to Mend APIs and, optionally, the Mend vulnerability database.
  • Proxy support: The Mend CLI and scanners support HTTP/HTTPS proxies with authentication for air-gapped or restricted environments.
  • Data sent to Mend: Generally includes dependency manifests, source code fingerprints for SAST, and container image layer metadata. No compiled artifacts or secrets are transmitted by default.
  • Self-hosted options: For highly regulated environments, discuss Mend self-hosted or on-premise scanning agents.

Key Concept: Data Flow Analysis

Data flow analysis documents where sensitive code, dependency metadata, and scan results move. Always trace the path from the developer's machine to the Mend platform, then to downstream systems like Jira or SIEM. Highlight encryption in transit, access controls, and retention policies.

Producing the Architecture Review Report

The final report turns the workshop into an actionable deliverable. It should be concise enough for executives but detailed enough for architects and security engineers.

  1. Executive Summary — One-page overview of current state, top gaps, and proposed Mend solution.
  2. Current-State Assessment — Tools, coverage, workflows, and pain points identified during discovery.
  3. Gap Analysis — Requirements mapped to current coverage and Mend capabilities.
  4. Future-State Architecture — Diagrams, integration points, network flows, and deployment phases.
  5. Roadmap and Next Steps — Pilot scope, success criteria, timeline, and owners.

Use the report to drive consensus among stakeholders. Share it before the next meeting so security, engineering, and DevOps leaders can align on scope and expectations. A well-written report often becomes the basis for the customer's internal business case.
