POC & Trial Management

🕐 40 min 📚 4 sections ⭐ 225 XP Level 2 - Intermediate

Designing Successful POCs

A Proof of Concept (POC) or technical trial is the most critical phase of an enterprise sales cycle. A well-designed POC validates Mend's value against the prospect's actual environment and pain points.

POC vs. Free Trial vs. Pilot

  • POC (2-3 weeks) — Focused evaluation with defined success criteria. Best for enterprise deals where technical validation drives the decision.
  • Free Trial (14-30 days) — Self-service exploration. Best for developer-led adoption and SMB customers.
  • Pilot (1-3 months) — Extended production deployment with a subset of repositories. Best for large enterprises needing to prove organizational readiness.

POC Planning & Execution

Pre-POC Preparation

POC Planning Checklist:

1. Success Criteria (agree BEFORE starting)
   [ ] Define 3-5 measurable outcomes
   [ ] Get written sign-off from economic buyer
   [ ] Set clear timeline with milestones

2. Technical Scope
   [ ] Identify 3-5 representative repositories
   [ ] Confirm tech stacks and languages
   [ ] Understand current tooling and pain points
   [ ] Map integration requirements (CI/CD, SCM, ticketing)

3. Stakeholders
   [ ] Security champion identified
   [ ] Engineering lead assigned
   [ ] 2-3 developers committed to participate
   [ ] Executive sponsor confirmed

4. Environment
   [ ] Mend organization provisioned
   [ ] SSO/SCIM configured (if required)
   [ ] CI/CD integration prepared
   [ ] IDE plugins ready for distribution

Common POC Scenarios

Scenario 1: "We have too many alerts from our current tool"

Setup: Scan the same repositories with both Mend and the current tool. Compare total findings vs. actionable findings.

Expected result: Mend's reachability analysis reduces actionable alerts by 60-80% while catching the same real vulnerabilities.

Scenario 2: "Our developers won't adopt security tools"

Setup: Install Mend IDE plugins for the developer participants. Track time-to-fix for Mend findings vs. their current workflow.

Expected result: Developers fix issues 5-10x faster with inline fix suggestions and one-click upgrades.

Scenario 3: "We need compliance-ready reporting"

Setup: Configure Mend to scan all POC repositories and generate SBOM + compliance reports.

Expected result: Automated report generation saves 100+ hours per audit cycle vs. manual evidence collection.

💡 POC Success Rule

Never start a POC without agreed-upon success criteria signed by the economic buyer. Without this, a "successful" POC can still result in no deal because stakeholders disagree on what success means.

POC Results Presentation

The results presentation is where you convert technical success into a purchase decision:

  1. Recap Success Criteria — Show each criterion and whether it was met
  2. Quantified Results — Use specific numbers: "Scanned 847 dependencies, found 23 critical reachable vulnerabilities, reduced noise by 78%"
  3. Developer Feedback — Share quotes and satisfaction scores from participating developers
  4. ROI Projection — Extrapolate POC results to the full organization
  5. Deployment Plan — Present a phased rollout plan for full organization deployment
  6. Commercial Proposal — If all criteria are met, present the commercial terms
