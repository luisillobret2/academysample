The Mend REST API and Authentication

The Mend REST API exposes the same data and controls available in the web UI, so anything you can see in the platform you can automate: pulling scan results, triggering reports, managing projects, and syncing findings into your own tools. Requests are made over HTTPS to your organization's API base URL, return JSON, and follow standard REST conventions with resource-oriented paths and HTTP verbs.

Authentication uses a short-lived bearer token. You exchange a long-lived organization or user API key for an access token, then send that token in the Authorization header on every subsequent request. Tokens expire, so a robust integration refreshes them automatically rather than hard-coding a single token. Never embed API keys in source code — load them from environment variables or a secrets manager.

The typical flow is: request a token, then use it until it expires, then request a new one. Here is a minimal example of logging in to obtain an access token and calling an authenticated endpoint:

# 1. Exchange your API key for a short-lived access token
curl -X POST "https://api.mend.io/api/v2.0/login" \
  -H "Content-Type: application/json" \
  -d '{
        "email": "svc-account@company.com",
        "orgToken": "$MEND_ORG_TOKEN",
        "userKey": "$MEND_USER_KEY"
      }'

# Response: { "retVal": { "jwtToken": "eyJhbGciOi..." } }

# 2. Call an authenticated endpoint with the token
curl "https://api.mend.io/api/v2.0/orgs/$ORG_UUID/projects" \
  -H "Authorization: Bearer eyJhbGciOi..." \
  -H "Content-Type: application/json"

💡 Security Tip

Create a dedicated service account with least-privilege scope for automation rather than using a human user's key. If a token leaks, you can revoke the service account without disrupting a person's access, and the audit trail clearly attributes automated actions.

Core Endpoints: Projects, Scans, Findings, and Reports

The API is organized around a small set of core resources. Projects represent the applications or repositories you scan and are the container for everything else. Scans are individual analysis runs against a project. Findings are the vulnerabilities, license issues, and policy violations a scan produces. Reports generate exportable artifacts such as SBOMs and compliance summaries.

A common automation walks from projects to findings: list the projects in an organization, then for each project retrieve its open findings and act on them. Findings support filtering by severity, status, and type so you can pull just the critical, unresolved items rather than the entire history. Reports are usually requested asynchronously — you request generation, then poll for completion and download the result.

# List open critical/high findings for a specific project
curl "https://api.mend.io/api/v2.0/projects/$PROJECT_UUID/findings?\
severity=critical,high&status=open&pageSize=100" \
  -H "Authorization: Bearer $TOKEN"

# Sample response (truncated)
{

    {

      "fixAvailable": true,
      "suggestedFix": "2.17.1"
    }
  ],

Pagination and Rate Limits

List endpoints return results in pages so a single request never has to serialize thousands of findings. You control paging with page and pageSize parameters, and the response reports the total item count so you can compute how many pages remain. Always page through to completion rather than assuming the first page contains everything — a large project can have hundreds of findings.

The API enforces rate limits to protect the platform. If you exceed the allowed request rate you will receive an HTTP 429 Too Many Requests response, often with a Retry-After header telling you how long to wait. Well-behaved integrations respect these limits by adding delays between requests and implementing exponential backoff on 429 responses instead of retrying immediately in a tight loop.

import os, time, requests

BASE = "https://api.mend.io/api/v2.0"
HEADERS = {"Authorization": f"Bearer {os.environ['MEND_TOKEN']}"}

def get_all_findings(project_uuid):
    findings, page, page_size = [], 0, 100
    while True:
        resp = requests.get(
            f"{BASE}/projects/{project_uuid}/findings",
            headers=HEADERS,
            params={"page": page, "pageSize": page_size, "status": "open"},
        )
        if resp.status_code == 429:                 # rate limited
            wait = int(resp.headers.get("Retry-After", 5))
            time.sleep(wait)                         # back off, then retry
            continue
        resp.raise_for_status()
        body = resp.json()
        findings.extend(body["retVal"])
        total = body["additionalData"]["totalItems"]
        page += 1
        if page * page_size >= total:               # last page reached
            break
    return findings

💡 Reliability Tip

Treat a 429 as normal, not exceptional. Build backoff into your client from day one — an integration that hammers the API and gets throttled mid-run is far harder to debug than one that paces itself and completes reliably.

Webhooks and Downstream Integrations

Polling the API on a schedule works, but webhooks let Mend push events to you the moment they happen, which is the right pattern for real-time reactions. You register an HTTPS endpoint and subscribe to event types; when a matching event occurs, Mend sends an HTTP POST with a JSON payload describing what changed. Common event types include scan completion, a new critical vulnerability discovered, and a policy violation raised.

Your webhook receiver should validate the payload signature to confirm it genuinely came from Mend, respond quickly with a 2xx status, and hand off any heavy work to a background queue so you never block the sender. From there you can fan out to downstream tools: open a Jira ticket for each new critical finding, post an alert to a Slack or Microsoft Teams channel, or update a custom dashboard.

from flask import Flask, request, abort
import hmac, hashlib, os, requests

app = Flask(__name__)
SECRET = os.environ["MEND_WEBHOOK_SECRET"].encode()

@app.post("/mend/webhook")
def mend_webhook():
    signature = request.headers.get("X-Mend-Signature", "")
    expected = hmac.new(SECRET, request.data, hashlib.sha256).hexdigest()
    if not hmac.compare_digest(signature, expected):
        abort(401)                                  # reject spoofed calls

    event = request.get_json()
    if event["type"] == "finding.new" and event["severity"] == "critical":
        create_jira_issue(event)                    # fan out to Jira
        notify_slack(event)                         # and to Slack
    return "", 204

def create_jira_issue(event):
    requests.post(
        "https://your-org.atlassian.net/rest/api/3/issue",
        auth=(os.environ["JIRA_USER"], os.environ["JIRA_TOKEN"]),
        json={"fields": {
            "project": {"key": "SEC"},
            "issuetype": {"name": "Bug"},
            "summary": f"[Mend] {event['name']} in {event['component']}",
        }},
    )

def notify_slack(event):
    requests.post(os.environ["SLACK_WEBHOOK_URL"], json={
        "text": f":rotating_light: New critical: {event['name']} "
                f"({event['component']}) fix: {event.get('suggestedFix','n/a')}"
    })
