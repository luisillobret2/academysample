Custom Security Policies for Developers

🕐 30 min 📚 4 sections ⭐ 175 XP Level 2 - Intermediate

Policy-as-Code for Security

Security policies should be defined as code, version-controlled, and enforced automatically. This ensures consistent security standards across all projects without manual gate-keeping that slows down development.

Policy Types

🐛

Vulnerability Policies

Define which severity levels block builds, which get warnings, and which are accepted risk.

📑

License Policies

Allowlist and blocklist licenses. Flag copyleft licenses, restrict AGPL, require legal review for unknown licenses.

📦

Package Policies

Block specific packages, enforce minimum versions, restrict packages based on age or reputation.

🔧

Code Quality Policies

SAST rule enforcement, custom code patterns to flag, and framework-specific security requirements.

Creating Custom Policies

# .mend/policy.yml
version: "1.0"

vulnerability_policies:
  - name: "Block Critical Vulnerabilities"
    severity: ["critical"]
    action: block
    message: "Critical vulnerabilities must be fixed before merge"

  - name: "Warn on High Vulnerabilities"
    severity: ["high"]
    action: warn
    message: "High vulnerabilities should be addressed within 7 days"

  - name: "Ignore Test Dependencies"
    scope:
      dependency_type: ["devDependencies", "test"]
    action: ignore

license_policies:
  - name: "Block Copyleft in Production"
    licenses: ["GPL-2.0", "GPL-3.0", "AGPL-3.0"]
    scope:
      dependency_type: ["dependencies"]
    action: block
    message: "Copyleft licenses not permitted in production dependencies"

  - name: "Allow Common Permissive"
    licenses: ["MIT", "Apache-2.0", "BSD-2-Clause", "BSD-3-Clause", "ISC"]
    action: allow

package_policies:
  - name: "Block Known Malicious"
    packages: ["event-stream@3.3.6", "ua-parser-js@0.7.29"]
    action: block

  - name: "Require Minimum Package Age"
    min_age_days: 7
    action: warn
    message: "New packages should be at least 7 days old"

Policy Inheritance & Overrides

Mend supports a hierarchical policy model where organization-wide policies provide the baseline and individual teams can add (but not relax) restrictions:

  • Organization Level — Global policies that apply to all repositories (e.g., block critical CVEs, no GPL)
  • Team/Group Level — Additional restrictions for specific teams (e.g., payment team blocks all dependencies older than 2 years)
  • Repository Level — Project-specific exceptions with documented justification and expiration dates

💡 Developer Tip

When a policy blocks your build, don't bypass it. Instead, file a policy exception request with a justification. Temporary exceptions can be granted with an expiration date, giving you time to find an alternative.

Working with Policy Exceptions

Sometimes legitimate business needs require temporary policy exceptions. Mend provides a structured exception workflow:

# .mend/exceptions.yml
exceptions:
  - id: "EXC-2025-001"
    description: "Temporary exception for legacy payment module migration"
    package: "xmlsec@1.3.5"
    vulnerability: "CVE-2024-12345"
    justification: "Not reachable in our code path; migrating to v2.0 in Q2"
    approved_by: "security-team@company.com"
    expires: "2025-06-30"
    jira_ticket: "SEC-1234"
  • Every Exception Needs a Ticket — Link to the tracking issue for remediation
  • Expiration Required — No permanent exceptions; maximum 90 days, renewable with review
  • Security Approval — All exceptions require sign-off from the security team
