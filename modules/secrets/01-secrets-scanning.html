Why Secrets Detection Matters

Leaked secrets — API keys, tokens, passwords, certificates, and private keys — are one of the most common and dangerous security vulnerabilities. According to industry research, over 10 million secrets are leaked in public repositories every year. A single exposed credential can give attackers access to databases, cloud infrastructure, payment systems, and customer data.

Mend.io's secrets detection capabilities help organizations identify hardcoded secrets across their entire codebase, including historical commits, to prevent credential-based breaches before they occur.

Types of Secrets Detected

Mend scans for a comprehensive set of secret types across your repositories:

  • API Keys & Tokens — AWS access keys, GCP service account keys, Azure connection strings, GitHub PATs, Slack tokens, Stripe keys
  • Database Credentials — Connection strings with embedded passwords for PostgreSQL, MySQL, MongoDB, Redis
  • Private Keys & Certificates — RSA/EC private keys, PEM files, PKCS12 keystores, SSH keys
  • Authentication Secrets — OAuth client secrets, JWT signing keys, SAML certificates, LDAP bind passwords
  • Infrastructure Secrets — Terraform state secrets, Kubernetes secrets in manifests, Docker registry credentials, CI/CD pipeline tokens

How Secrets Scanning Works

Mend uses a multi-layered detection approach combining pattern matching, entropy analysis, and verification to minimize false positives while maximizing detection coverage:

🔎

Pattern Matching

Over 700 regex patterns for known secret formats from cloud providers, SaaS platforms, and common frameworks.

📈

Entropy Analysis

Shannon entropy scoring identifies high-randomness strings that match the statistical profile of credentials.

✅

Live Verification

Where possible, Mend verifies if detected secrets are still active by safely testing them against provider APIs.

🕑

Historical Scanning

Scans the full git history — not just current files — to find secrets that were committed and later removed.

Configuring Secrets Scanning

# Enable secrets scanning via Mend CLI
mend secrets scan --dir ./my-project

# Scan including git history
mend secrets scan --dir ./my-project --history

# Scan with specific severity threshold
mend secrets scan --dir ./my-project --min-severity high

# Output results in SARIF format for CI integration
mend secrets scan --dir ./my-project --format sarif --output results.sarif

Integration with CI/CD Pipelines

Secrets scanning should be integrated into your development workflow to catch leaked credentials before they reach production:

Pre-commit Hook

Block secrets from being committed in the first place:

# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: mend-secrets
        name: Mend Secrets Scan
        entry: mend secrets scan --dir . --fail-on high
        language: system
        pass_filenames: false

GitHub Actions Integration

# .github/workflows/secrets-scan.yml
name: Secrets Scan
on: [push, pull_request]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for historical scanning
      - name: Run Mend Secrets Scan
        run: |
          mend secrets scan --dir . --history --format sarif --output results.sarif
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: results.sarif

💡 Best Practice

Always use fetch-depth: 0 in your checkout step to enable historical scanning. Secrets removed from current files may still exist in git history and can be extracted by attackers.
