Centralized Secrets Management

Secrets managers provide a centralized, encrypted store for all credentials, eliminating the need to hardcode secrets in configuration files, environment variables, or source code. They serve as the single source of truth for credential distribution.

Integrating Mend with Vault

Mend.io detects when applications reference secrets managers and validates that the integration follows security best practices. The scanner identifies anti-patterns like caching vault tokens or using overly broad access policies.

Vault Agent Sidecar Pattern

# Kubernetes deployment with Vault Agent sidecar
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  template:
    metadata:
      annotations:
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "myapp"
        vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/readonly"
        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/readonly" -}}
          postgresql://{{ .Data.username }}:{{ .Data.password }}@db:5432/mydb
          {{- end -}}
    spec:
      serviceAccountName: myapp
      containers:
        - name: myapp
          image: myapp:latest
          volumeMounts:
            - name: vault-secrets
              mountPath: /vault/secrets
              readOnly: true

Access Control & Audit

Proper access control is critical for secrets management. Every secret access should follow the principle of least privilege:

  • Identity-Based Access — Use service identities (Kubernetes service accounts, AWS IAM roles) rather than static tokens
  • Time-Bound Access — Dynamic secrets with short TTLs ensure credentials expire automatically
  • Audit Logging — Every secret read, write, and rotation event should be logged for compliance
  • Path-Based Policies — Organize secrets by environment and service, with policies restricting access to relevant paths only

Vault Policy Example

# Policy for production database team
path "secret/data/production/database/*" {
  capabilities = ["read", "list"]
}

path "database/creds/production-readonly" {
  capabilities = ["read"]
}

# Deny access to other environments
path "secret/data/staging/*" {
  capabilities = ["deny"]
}

Migration from Hardcoded Secrets

Migrating from hardcoded secrets to a vault is a multi-step process. Mend can identify all hardcoded secrets to create a migration plan:

  1. Discovery — Use Mend secrets scanning to inventory all hardcoded secrets across your repositories
  2. Classification — Categorize secrets by sensitivity level and the services that depend on them
  3. Vault Setup — Create the appropriate secret engines, paths, and access policies in your vault
  4. Migration — Move secrets to the vault one service at a time, updating application code to use vault references
  5. Verification — Re-run Mend scanning to confirm no hardcoded secrets remain
  6. Prevention — Enable pre-commit hooks and CI pipeline checks to block future hardcoded secrets

💡 Pro Tip

Start vault migration with non-production environments. This lets your team build confidence with the new workflow before tackling production secrets.
