Secrets Rotation & Lifecycle

🕐 35 min 📚 4 sections ⭐ 175 XP Level 2 - Intermediate

The Importance of Secrets Rotation

Even with the best scanning in place, secrets have a natural lifecycle. API keys, tokens, and credentials should be rotated regularly to limit the blast radius of any potential exposure. Industry best practices recommend rotating most secrets every 90 days or less.

Mend.io helps organizations track secret age and enforce rotation policies across their infrastructure, ensuring that stale credentials don't become attack vectors.

Why Regular Rotation Matters

  • Limits Exposure Window — If a secret is leaked, regular rotation ensures it becomes invalid quickly
  • Compliance Requirements — PCI DSS, SOC 2, and HIPAA all require credential rotation policies
  • Reduces Insider Risk — Former employees or contractors lose access as credentials rotate
  • Audit Trail — Rotation events create a clear audit log for security reviews

Automated Rotation Strategies

Manual rotation is error-prone and doesn't scale. Modern organizations use automated rotation to ensure consistent, reliable credential management:

🔄

Scheduled Rotation

Configure automatic rotation at fixed intervals (30, 60, or 90 days) for all managed credentials.

🚨

Event-Triggered Rotation

Automatically rotate secrets when a leak is detected, an employee leaves, or a security event occurs.

🔗

Zero-Downtime Rotation

Dual-secret patterns allow new credentials to be deployed before old ones are revoked, preventing outages.

📊

Rotation Reporting

Dashboards track rotation compliance, flag overdue credentials, and generate audit reports.

Implementing Rotation with Cloud Providers

Major cloud providers offer native rotation capabilities that integrate with Mend's scanning:

AWS Secrets Manager Rotation

# AWS Lambda rotation function
import boto3
import json

def lambda_handler(event, context):
    secret_id = event['SecretId']
    step = event['Step']
    client = boto3.client('secretsmanager')

    if step == 'createSecret':
        # Generate new secret value
        new_password = client.get_random_password(
            PasswordLength=32,
            ExcludeCharacters='/@"\''
        )
        client.put_secret_value(
            SecretId=secret_id,
            ClientRequestToken=event['ClientRequestToken'],
            SecretString=json.dumps({'password': new_password['RandomPassword']}),
            VersionStages=['AWSPENDING']
        )

# Create a role with automatic expiration
vault write database/roles/readonly \
    db_name=mydb \
    creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}'; GRANT SELECT ON *.* TO '{{name}}'@'%';" \
    default_ttl="1h" \
    max_ttl="24h"

Rotation Policy Best Practices

A comprehensive rotation policy should cover the following areas:

  • Classification Tiers — High-risk secrets (production DB passwords, payment keys) rotate every 30 days; standard secrets every 90 days
  • Break-Glass Procedures — Define emergency rotation processes for when a breach is suspected
  • Dependency Mapping — Track which services use each secret to coordinate rotation across systems
  • Testing Requirements — Verify that rotated secrets work in staging before production deployment
  • Notification Chains — Alert relevant teams before and after rotation events

💡 Pro Tip

Start with your highest-risk secrets first. Database passwords and cloud provider keys should be the first candidates for automated rotation. Expand to other secret types as your rotation infrastructure matures.
