Secret Leak Incident Response

When a secret is detected in your codebase, time is critical. The average time between a secret being leaked and an attacker exploiting it can be as little as minutes. Mend provides a structured incident response workflow to help teams act quickly and decisively.

The 5-Step Response Process

  1. Identify & Classify — Determine the type of secret, its scope (which systems it grants access to), and whether it's been verified as active
  2. Revoke Immediately — Rotate or disable the compromised credential at its source (cloud provider, SaaS platform, database). Do this before removing it from code
  3. Assess Exposure — Determine when the secret was first committed, which branches contain it, and whether the repository is public or private
  4. Remediate in Code — Remove the secret from all branches, including git history if the repo is public. Replace with environment variables or a secrets manager
  5. Audit & Monitor — Check access logs for the compromised credential to determine if unauthorized access occurred. Set up monitoring for the replacement credential

⚠️ Critical Warning

Never just delete a leaked secret from your code and assume the problem is solved. The secret exists in git history and may already be compromised. Always revoke first, then remediate.

Using Secrets Managers

The best remediation for hardcoded secrets is to move them to a dedicated secrets management solution. Mend integrates with and recommends the following approaches:

☁️

Cloud Provider Vaults

AWS Secrets Manager, Azure Key Vault, GCP Secret Manager — native solutions with automatic rotation.

🔑

HashiCorp Vault

Platform-agnostic secrets management with dynamic secrets, encryption as a service, and identity-based access.

💻

Environment Variables

The simplest approach: inject secrets at runtime via CI/CD pipeline variables or container orchestration.

🔐

.env Files + .gitignore

For local development: use .env files that are excluded from version control via .gitignore rules.

Migration Example

# BEFORE: Hardcoded secret in code
db_connection = "postgresql://admin:SuperSecret123@db.example.com:5432/production"

# AFTER: Using environment variable
import os
db_connection = os.environ.get("DATABASE_URL")

# AFTER: Using AWS Secrets Manager
import boto3
client = boto3.client('secretsmanager')
secret = client.get_secret_value(SecretId='prod/database/credentials')
db_connection = json.loads(secret['SecretString'])['connection_url']

Policy Enforcement

Mend allows you to create organizational policies that automate secrets detection and enforcement:

  • Block PR Merge — Automatically block pull requests that contain detected secrets from being merged
  • Alert & Notify — Send Slack/Teams notifications when secrets are detected in any repository
  • Auto-Revoke — For supported providers, automatically trigger credential rotation when a leak is detected
  • Compliance Reporting — Generate reports showing secrets detection coverage across all repositories for audit purposes
