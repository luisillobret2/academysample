Understanding SBOM Formats

Software Bills of Materials describe the components, licenses, and dependencies in an application. The two most common formats are CycloneDX and SPDX.

# SBOM format comparison CycloneDX: Lightweight, security-focused, easy to integrate SPDX: ISO standard, comprehensive license and provenance data
$ mend sbom formats list Supported formats: - CycloneDX 1.4 - CycloneDX 1.5 - SPDX 2.3 - SPDX 2.2
Compliance context Executive Order 14028 and the EU Cyber Resilience Act require organizations to produce and share SBOMs for software components.
  • ✓ Compared CycloneDX and SPDX
  • ✓ Identified compliance requirements for your SBOM
2

Generate SBOM from a Scan

Use the Mend CLI to export an SBOM directly from your SCA scan results. CycloneDX is the default format and is widely supported by security tools.

# Generate CycloneDX SBOM mend sbom export --format cyclonedx --output sbom.json
$ mend sbom export --format cyclonedx --output sbom.json Generating SBOM... Format: CycloneDX 1.5 Components: 183 Dependencies: 47 SBOM written to sbom.json (42.3 KB)
Tip Run the SBOM export after every successful scan so the exported file reflects the current dependency tree.
  • ✓ Generated SBOM from the scan results
  • ✓ Verified the component count
  • ✓ Inspected the sbom.json file
3

Enrich SBOM with Vulnerability Data

Add VEX (Vulnerability Exploitability eXchange) data to the SBOM to communicate whether known vulnerabilities are exploitable and how they are being handled.

# Generate SBOM with VEX data mend sbom export --format cyclonedx --include-vex --output sbom-vex.json
$ mend sbom export --format cyclonedx --include-vex --output sbom-vex.json Generating SBOM with VEX annotations... Format: CycloneDX 1.5 Components: 183 VEX statements: 27 Not affected: 15 Affected: 8 Fixed: 4 SBOM written to sbom-vex.json (48.7 KB)
Key concept VEX helps reduce noise by documenting why a vulnerable component is not exploitable or how it has been remediated.
  • ✓ Generated SBOM with VEX data
  • ✓ Reviewed VEX statements for accuracy
4

Automate SBOM Generation in CI/CD

Create a GitHub Actions workflow that generates and attaches an SBOM to every release. This ensures SBOMs are always available for compliance and distribution.

# GitHub Actions: generate SBOM on release name: Generate SBOM on: release: types: [published] jobs: sbom: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Generate SBOM run: mend sbom export --format cyclonedx --output sbom.json - name: Upload SBOM to release uses: softprops/action-gh-release@v2 with: files: sbom.json
$ mend ci sbom --pipeline github Detected GitHub Actions workflow Adding SBOM generation step to release workflow CI/CD integration configured.
Tip Store SBOMs as release artifacts and version them alongside your application so consumers can retrieve the SBOM for any deployed version.
  • ✓ Added SBOM generation step to release workflow
  • ✓ Configured SBOM upload as a release artifact
5

Validate and Distribute SBOM

Validate the SBOM structure before sharing it with stakeholders. Then distribute the SBOM to compliance tools, customers, and internal teams.

# Validate SBOM structure mend sbom validate --file sbom.json # Distribute SBOM to stakeholders mend sbom share --file sbom.json --recipients "security@example.com,compliance@example.com"
$ mend sbom validate --file sbom.json Validating SBOM structure... Format: CycloneDX 1.5 Components: 183 SBOM is valid. Warnings: 0 Errors: 0
Tip Schedule SBOM validation as a quality gate in CI/CD so that only valid SBOMs are published or distributed.
  • ✓ Validated SBOM with no errors
  • ✓ Distributed SBOM to stakeholders
  • ✓ Integrated SBOM into compliance tooling
