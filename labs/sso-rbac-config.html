SSO & RBAC Configuration

Guided
🕓 60 min 🏆 100 XP 🔄 Administration 📚 5 steps

🎯 Lab Objectives

  • Configure SAML 2.0 single sign-on for Mend
  • Map identity provider attributes to Mend user fields
  • Create a custom role hierarchy aligned with your teams
  • Assign roles to teams based on SSO group membership
  • Verify SSO login flows and role-based access restrictions
1

Configure SAML SSO

Set up a SAML 2.0 identity provider such as Okta or Azure AD, then configure Mend as the service provider. Download your IdP metadata XML and upload it to Mend.

# Configure Mend SSO using IdP metadata mend sso configure --idp-metadata metadata.xml --entity-id mend-academy
$ mend sso configure --idp-metadata metadata.xml --entity-id mend-academy Reading IdP metadata from metadata.xml... Entity ID: mend-academy SSO URL: https://idp.example.com/saml/sso SAML SSO configuration saved successfully.
Tip Keep a copy of the Mend service provider metadata to upload into your IdP. This ensures the audience and assertion consumer URLs match.
  • ✓ Uploaded IdP metadata XML to Mend
  • ✓ Configured entity ID and SSO URL
  • ✓ Saved Mend SP metadata in the IdP
2

Map User Attributes

Map SAML attributes from your identity provider to Mend user fields. This ensures users are created with the correct email, display name, and group membership.

# Attribute mapping configuration Email: user.email Name: user.name Groups: user.groups
$ mend sso attributes map --email user.email --name user.name --groups user.groups Mapping SAML attributes to Mend user fields... Email attribute mapped to user.email Name attribute mapped to user.name Groups attribute mapped to user.groups Attribute mapping saved.
Tip Use the same attribute names configured in your IdP. Group values are used to determine role assignments in the next step.
  • ✓ Mapped email, name, and group attributes
  • ✓ Verified attribute names in the IdP
3

Create Role Hierarchy

Define custom roles that match your organization's responsibilities. Start with an Admin role, then add Security Lead, Developer, and Viewer roles with scoped permissions.

# Create custom roles mend roles create --name "Admin" --permissions "all" mend roles create --name "Security Lead" --permissions "view,triage,policy" mend roles create --name "Developer" --permissions "view,triage" mend roles create --name "Viewer" --permissions "view"
$ mend roles create --name "Security Lead" --permissions "view,triage,policy" Creating role Security Lead... Role Security Lead created with permissions: view, triage, policy Role ID: role-sec-lead-001
Notice Grant the minimum permissions needed for each role. Admin can manage users and policies, while Viewer is read-only.
  • ✓ Created Admin, Security Lead, Developer, and Viewer roles
  • ✓ Verified each role has the correct permissions
  • ✓ Documented role responsibilities
4

Assign Roles to Teams

Map SSO groups to Mend roles so users inherit the correct access automatically. Assign the security-team group to the Security Lead role and development-team to the Developer role.

# Assign SSO groups to roles mend teams assign --group "security-team" --role "Security Lead" mend teams assign --group "development-team" --role "Developer" mend teams assign --group "executives" --role "Viewer"
$ mend teams assign --group "security-team" --role "Security Lead" Assigning group security-team to role Security Lead... Team assignment created. Users in security-team will inherit the Security Lead role on next login.
Tip Review assignments before enabling SSO enforcement. Users without a mapped group may be denied access or assigned a default role.
  • ✓ Mapped each SSO group to a Mend role
  • ✓ Confirmed team assignments are active
5

Test SSO Login & Access

Verify the SSO login flow and confirm that role-based access restrictions are enforced. A user in the Developer role should not be able to edit policies.

# Test SSO login for a developer mend sso test --user developer@example.com --role Developer
$ mend sso test --user developer@example.com --role Developer Initiating SSO login for developer@example.com... SSO login successful Role assigned: Developer Testing policy edit permission... Access denied: Developer role cannot edit policies. Role-based access restriction verified.
Key concept RBAC relies on least privilege. Test both positive and negative cases to ensure users can perform allowed actions and are blocked from restricted ones.
  • ✓ Successfully logged in via SSO
  • ✓ Verified role assignment on login
  • ✓ Confirmed Developer cannot edit policies

Which SAML attribute is typically used to map users to Mend roles?

You have configured SAML SSO, mapped user attributes, created a role hierarchy, assigned teams, and verified role-based access.
