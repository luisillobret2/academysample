Secrets Detection Setup

Guided
🕓 45 min 🏆 100 XP 🔄 Administration 📚 5 steps

🎯 Lab Objectives

  • Enable Mend secrets detection for a project
  • Configure custom detection rules for internal patterns
  • Run a historical scan of git history for leaked secrets
  • Set up incident response alerting and webhooks
  • Remediate detected secrets by rotating and cleaning git history
1

Enable Secrets Scanning

Turn on Mend secrets detection for your project and configure the scan scope. This activates scanning across all configured repositories for the project.

# Enable secrets detection for a project mend secrets enable --project "Lab: Java App" --scan-all-repos # Verify secrets scanning is active mend secrets status --project "Lab: Java App"
$ mend secrets enable --project "Lab: Java App" --scan-all-repos Enabling secrets detection for project: Lab: Java App Scan scope: all repositories (3 detected) Default ruleset: Mend Built-in (78 patterns) Secrets detection enabled successfully. $ mend secrets status --project "Lab: Java App" Status: ACTIVE Repositories scanned: 3 Last scan: never
Tip The --scan-all-repos flag ensures every repository linked to the project is included. Use --scan-scope branch:main to limit scanning to specific branches.
  • ✓ Secrets detection enabled for the project
  • ✓ Verified scanning status reports ACTIVE
  • ✓ Confirmed all repositories are in scan scope
2

Configure Detection Rules

Set up custom secret patterns to catch organization-specific credentials such as internal API keys, tokens, and proprietary formats that built-in rules do not cover.

# Create a custom rule for AWS access keys mend secrets rules create --name "Internal API Key" --pattern "AKIA[0-9A-Z]{16}" --severity critical # Create a custom rule for internal auth tokens mend secrets rules create --name "Internal Auth Token" --pattern "mnt_[a-zA-Z0-9]{40}" --severity high # List all configured rules mend secrets rules list
$ mend secrets rules create --name "Internal API Key" --pattern "AKIA[0-9A-Z]{16}" --severity critical Rule created: Internal API Key (ID: rule_8f3a) Pattern: AKIA[0-9A-Z]{16} Severity: critical $ mend secrets rules list Built-in rules: 78 Custom rules: 2 Internal API Key critical rule_8f3a Internal Auth Token high rule_2c71
Notice Regex patterns are matched against file contents and commit diffs. Avoid overly broad patterns that generate false positives. Test new rules with mend secrets rules test before enabling in production.
  • ✓ Created the Internal API Key rule with critical severity
  • ✓ Created the Internal Auth Token rule with high severity
  • ✓ Verified both custom rules appear in the rules list
3

Run a Historical Scan

Scan the full git history to find secrets leaked in past commits. A historical scan examines every commit up to the specified depth, surfacing credentials that may have been committed and forgotten.

# Scan git history for leaked secrets (up to 500 commits deep) mend secrets scan --historical --depth 500 # Export findings for review mend secrets findings export --format json --output secrets-findings.json
$ mend secrets scan --historical --depth 500 Starting historical scan... Repository: java-vulnerable-app Scanning 500 commits... Scanning commit a3f1c2d (2023-06-14) SECRET FOUND: Internal API Key File: src/main/resources/config.properties Line: 12 Value: AKIAIOSFODNN7EXAMPLE* Scanning commit 8b9e021 (2023-08-02) SECRET FOUND: AWS Access Key File: .env.example Line: 5 Historical scan complete. Critical: 2 secrets found High: 1 secret found Total: 3 secrets across 2 commits
Notice The scan found 3 secrets in old commits. Even though these may no longer be in the current branch, the credentials remain in git history and must be treated as compromised.
  • ✓ Historical scan completed across 500 commits
  • ✓ Reviewed all 3 detected secrets and their locations
  • ✓ Exported findings to secrets-findings.json
4

Set Up Incident Response

Configure alerting so that detected secrets trigger immediate notifications. Mend supports Slack, email, and Jira ticket creation so the right team is notified the moment a secret is found.

# Configure Slack alerting mend alerts configure --channel slack --webhook "https://hooks.slack.com/services/T0/B0/XXXX" --severity critical,high # Configure email alerting mend alerts configure --channel email --recipients "security@mend.io,oncall@mend.io" --severity critical # Configure Jira ticket creation for critical secrets mend alerts configure --channel jira --project "SEC" --issue-type Incident --severity critical
$ mend alerts configure --channel slack --webhook "https://hooks.slack.com/services/T0/B0/XXXX" --severity critical,high Slack alerting configured. Channel: #security-alerts Severities: critical, high $ mend alerts configure --channel jira --project "SEC" --issue-type Incident --severity critical Jira integration configured. Project: SEC Issue type: Incident A ticket will be created for each critical secret detection.
Tip Route critical secrets to multiple channels simultaneously. The Jira integration auto-creates a tracked incident ticket so remediation status is visible to the security team end-to-end.
  • ✓ Slack webhook alerting configured for critical and high secrets
  • ✓ Email recipients set for critical severity detections
  • ✓ Jira ticket creation enabled for critical incidents
5

Remediate Detected Secrets

Follow the incident response workflow for each detected secret: rotate the compromised credential, remove it from git history, and verify the remediation is complete.

# Step 1: Rotate the compromised secret in the provider console # (e.g. AWS IAM console - deactivate and create a new key) # Step 2: Remove the secret from git history using git-filter-repo pip install git-filter-repo git filter-repo --replace-text <(echo "AKIAIOSFODNN7EXAMPLE==>REDACTED") # Step 3: Force-push the cleaned history git push origin --force --all # Step 4: Re-scan to verify remediation mend secrets scan --historical --depth 500
$ git filter-repo --replace-text <(echo "AKIAIOSFODNN7EXAMPLE==>REDACTED") Parsing commits... Rewriting 2 commits... Replaced 1 occurrence in 1 commit. Reflog and origin remotes removed. $ git push origin --force --all Enumerating objects: 487, done. Force-pushed rewritten history to origin. $ mend secrets scan --historical --depth 500 Re-scanning git history... Scan complete. 0 secrets found. Remediation verified successfully.
Notice Rotating the secret is mandatory. Removing it from git history does not undo the exposure - anyone who cloned the repo before the rewrite still has the original credential. Always rotate first, then clean history.
  • ✓ Rotated the compromised credential in the provider console
  • ✓ Removed the secret from git history and force-pushed
  • ✓ Re-scanned and confirmed 0 secrets remaining
