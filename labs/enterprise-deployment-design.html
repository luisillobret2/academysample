Enterprise Deployment Design

Challenge
🕓 120 min 🏆 300 XP 🔄 Administration 📚 5 steps

🎯 Lab Objectives

  • Analyze enterprise requirements from a customer scenario
  • Design a multi-org hierarchy with business unit separation
  • Configure organization-level and project-level security policies
  • Implement a templated CI/CD approach for 50 repositories
  • Document a phased deployment plan with metrics and risk mitigation
1

Analyze Enterprise Requirements

Review the customer scenario before designing the deployment. Understanding the scale, organizational structure, and compliance obligations drives every downstream architecture decision.

# Open the customer scenario brief cat ~/labs/enterprise-scenario.md # Summarize key requirements mend academy scenario --summarize --file enterprise-scenario.md
$ mend academy scenario --summarize --file enterprise-scenario.md Customer: Global Financial Services Corp Key Requirements: Developers: 500 across 3 business units Repositories: 50 (Java, Python, Go, Node.js) Business units: Retail Banking, Investment, Corporate IT SSO: Required (Okta SAML) Compliance: SOC 2 Type II, PCI-DSS CI/CD: GitHub Actions + Jenkins Priorities: Reachability analysis, AI remediation Constraints: No secrets in shared repos across BUs PCI-DSS scope isolation for cardholder data repos Audit reporting per business unit quarterly
Notice The PCI-DSS requirement for cardholder data isolation is a hard constraint. Repos handling cardholder data must be in a separate organization with strict policy enforcement and quarterly audit reporting.
  • ✓ Reviewed the customer scenario and summarized all key requirements
  • ✓ Identified PCI-DSS scope isolation as a hard architectural constraint
  • ✓ Mapped the 50 repositories to their respective business units
2

Design the Org Structure

Create a multi-org hierarchy that separates business units and isolates PCI-DSS scope. Map teams to organizations and assign roles following least-privilege principles.

# Create the top-level enterprise organizations mend org create --name "GFSC Retail Banking" --parent "Global Financial Services Corp" mend org create --name "GFSC Investment" --parent "Global Financial Services Corp" mend org create --name "GFSC Corporate IT" --parent "Global Financial Services Corp" mend org create --name "GFSC PCI-Scope" --parent "Global Financial Services Corp" --isolated # Assign teams and roles mend team create --org "GFSC Retail Banking" --name "retail-apps" --repos 18 mend team create --org "GFSC Investment" --name "trading-platforms" --repos 15 mend team create --org "GFSC Corporate IT" --name "internal-tools" --repos 12 mend team create --org "GFSC PCI-Scope" --name "cardholder-data" --repos 5
$ mend org create --name "GFSC PCI-Scope" --parent "Global Financial Services Corp" --isolated Organization created: GFSC PCI-Scope Isolation: enabled (no cross-org data sharing) Audit reporting: quarterly (PCI-DSS requirement) $ mend team create --org "GFSC PCI-Scope" --name "cardholder-data" --repos 5 Team created: cardholder-data (5 repos) Default role: Developer (least privilege) Org Structure: Global Financial Services Corp ├─ GFSC Retail Banking [18 repos, 180 dev] ├─ GFSC Investment [15 repos, 150 dev] ├─ GFSC Corporate IT [12 repos, 120 dev] └─ GFSC PCI-Scope (ISO) [ 5 repos, 50 dev]
Tip Use role-based access control with least privilege. Only the security team gets Admin roles; developers get Developer roles with scan access but no policy modification rights. Assign Auditors read-only access to compliance reports.
  • ✓ Created 4 organizations with PCI-Scope isolated
  • ✓ Mapped all 50 repositories across the 4 organizations
  • ✓ Assigned teams and roles following least-privilege principles
3

Configure Security Policies

Set up organization-level policies with project-level overrides where needed. Apply compliance frameworks so scan results map directly to SOC 2 and PCI-DSS control requirements.

# Set the enterprise-wide baseline policy mend policy create --org "Global Financial Services Corp" --name "Enterprise Baseline" \ --block-critical true --block-high true --fail-on-license-violation true # Apply compliance frameworks mend policy framework add --org "GFSC PCI-Scope" --framework pci-dss mend policy framework add --org "GFSC Retail Banking" --framework soc2 # Create a project-level override for a legacy system mend policy override --org "GFSC Corporate IT" --project "legacy-billing" \ --block-critical false --expires "2026-12-31" --reason "Legacy system, remediation in Q4"
$ mend policy create --org "Global Financial Services Corp" --name "Enterprise Baseline" \ --block-critical true --block-high true --fail-on-license-violation true Policy created: Enterprise Baseline Scope: all child organizations Block on critical: yes Block on high: yes Fail on license violation: yes $ mend policy framework add --org "GFSC PCI-Scope" --framework pci-dss PCI-DSS framework applied to GFSC PCI-Scope. Mapped controls: 24 Quarterly audit report: enabled $ mend policy override --org "GFSC Corporate IT" --project "legacy-billing" \ --block-critical false --expires "2026-12-31" --reason "Legacy system, remediation in Q4" Override created with expiration date. Project: legacy-billing Expires: 2026-12-31 (will auto-revert)
Notice Policy overrides must always have an expiration date and a documented reason. The PCI-Scope organization should have zero overrides - compliance requirements do not allow exceptions for cardholder data systems.
  • ✓ Created the enterprise baseline policy blocking critical and high vulnerabilities
  • ✓ Applied PCI-DSS and SOC 2 frameworks to the appropriate organizations
  • ✓ Created a time-boxed project override with expiration and reason

Implement CI/CD at Scale

Design a templated CI/CD approach so all 50 repositories scan consistently without per-repo customization. Create a reusable GitHub Actions workflow and a Jenkins shared library.

# .github/workflows/mend-scan.yml (reusable workflow) name: Mend Security Scan on: [pull_request, push] jobs: mend-scan: uses: mend-io/mend-templates/.github/workflows/mend-scan.yml@v2 with: scan-type: sca+sast fail-on: critical,high secrets: MEND_API_KEY: ${{ secrets.MEND_API_KEY }} # Jenkins shared library: vars/mendScan.groovy def call(Map config = [:]) { mendScan( scanType: config.scanType ?: 'sca+sast', failOn: config.failOn ?: 'critical,high', orgToken: env.MEND_ORG_TOKEN ) }
$ mend ci template validate --file .github/workflows/mend-scan.yml Template valid. Reusable across 50 repos. Scan type: sca+sast Fail threshold: critical, high Secrets required: MEND_API_KEY $ mend ci apply --template mend-scan.yml --repos 50 Applying template to 50 repositories... GitHub Actions: 35 repos configured Jenkins shared lib: 15 repos configured All 50 repositories configured with Mend scanning. Estimated scan time per repo: 2-4 minutes
Tip Centralize the workflow in a templates repository and reference it with uses: so updates propagate to all repos automatically. Never copy-paste scan configs into individual repos - that creates maintenance debt at scale.
  • ✓ Created a reusable GitHub Actions workflow in a templates repository
  • ✓ Created a Jenkins shared library for the Jenkins-based repos
  • ✓ Applied the template to all 50 repositories successfully
5

Create the Deployment Plan

Document a phased rollout plan with timelines, success metrics, and risk mitigation. A three-phase approach (pilot, expand, full deployment) limits blast radius and builds organizational confidence.

# Generate the deployment plan document mend academy plan generate --scenario enterprise-scenario.md --output deployment-plan.md # Review the phased rollout timeline cat deployment-plan.md
$ cat deployment-plan.md Enterprise Deployment Plan - Global Financial Services Corp Phase 1: Pilot (Weeks 1-3) Scope: 1 BU (Corporate IT), 12 repos, 120 dev Goals: Validate scanning, tune policies, train champions Success: 90% repos scanning, <5% false positive rate Risk: Policy too strict blocks CI - mitigation: advisory mode week 1 Phase 2: Expand (Weeks 4-8) Scope: +2 BUs (Retail Banking, Investment), +33 repos, +330 dev Goals: Enforce blocking policies, AI remediation adoption Success: 95% repos scanning, 50% critical fixes via AI remediation Risk: Developer friction - mitigation: office hours, runbooks Phase 3: Full Deployment (Weeks 9-12) Scope: +PCI-Scope org, +5 repos, +50 dev, SSO + compliance live Goals: PCI-DSS audit reporting, SOC 2 evidence collection Success: 100% repos scanning, quarterly audit report generated Risk: PCI audit findings - mitigation: pre-audit scan, remediation sprint
Notice Start Phase 1 in advisory mode (report-only, no CI blocking) for the first week. This lets developers see findings without build failures, reducing friction before enforcing blocking policies.
  • ✓ Documented a 3-phase rollout with clear timelines (12 weeks total)
  • ✓ Defined measurable success metrics for each phase
  • ✓ Identified key risks and documented mitigation strategies per phase
