GitHub Actions Integration

Guided
🕓 60 min 🏆 100 XP 🔄 Integration 📚 6 steps

🎯 Lab Objectives

  • Fork a sample repository and configure Mend API secrets in GitHub
  • Create a GitHub Actions workflow that runs the Mend scan action
  • Configure SCA + SAST scan parameters and severity-based failure conditions
  • Enable automated pull request comments with scan results
  • Trigger the pipeline and verify scan results as PR feedback
  • Set up branch protection rules and policy-based quality gates
1

Prerequisites & Repository Setup

Start by forking the sample repository and storing your Mend API credentials as encrypted GitHub Actions secrets. These secrets are injected into the workflow at runtime and never exposed in logs.

# Fork the sample repository gh repo fork mend-academy/sample-node-app --clone # Navigate into the cloned repo cd sample-node-app
$ gh repo fork mend-academy/sample-node-app --clone Cloning into 'sample-node-app'... Forked to: your-org/sample-node-app $ cd sample-node-app

Open your forked repository on GitHub, then navigate to Settings → Secrets and variables → Actions and add the following secrets:

  • MEND_EMAIL — Your Mend account email
  • MEND_KEY — Your Mend API key from app.mend.io/settings/api-keys
  • MEND_URL — Your Mend tenant URL (e.g. https://app.mend.io)
Important Never commit your API key to the repository. Secrets stored in GitHub Actions are encrypted at rest and only exposed to workflows that explicitly reference them.
  • ✓ Forked the sample repository to your account
  • ✓ Added MEND_EMAIL, MEND_KEY, and MEND_URL as repository secrets
2

Create the GitHub Actions Workflow

Create a new workflow file at .github/workflows/mend-scan.yml. This workflow will trigger on every push and pull request, run the official mend-action/scan action, and upload results back to the Mend platform.

# Create the workflows directory if it does not exist mkdir -p .github/workflows # Create the mend-scan.yml file touch .github/workflows/mend-scan.yml

Add the following YAML content to .github/workflows/mend-scan.yml:

# .github/workflows/mend-scan.yml name: Mend Security Scan on: push: branches: [main, develop] pull_request: branches: [main] jobs: mend-scan: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Run Mend Scan uses: mend-action/scan@v1 with: mend-email: ${{ secrets.MEND_EMAIL }} mend-key: ${{ secrets.MEND_KEY }} mend-url: ${{ secrets.MEND_URL }} type: "sca,sast"
Tip The ${{ secrets.X }} syntax references the repository secrets you created in Step 1. GitHub automatically masks these values in the workflow logs.
  • ✓ Created .github/workflows/mend-scan.yml
  • ✓ Workflow triggers on push to main/develop and on pull requests
3

Configure Scan Parameters

Customize the workflow to run both SCA and SAST scans, and configure a failures policy that breaks the build when critical-severity findings are detected. This prevents vulnerable code from being merged.

# Updated .github/workflows/mend-scan.yml name: Mend Security Scan on: push: branches: [main, develop] pull_request: branches: [main] jobs: mend-scan: runs-on: ubuntu-latest permissions: contents: read pull-requests: write steps: - name: Checkout code uses: actions/checkout@v4 - name: Run Mend SCA and SAST Scan uses: mend-action/scan@v1 with: mend-email: ${{ secrets.MEND_EMAIL }} mend-key: ${{ secrets.MEND_KEY }} mend-url: ${{ secrets.MEND_URL }} type: "sca,sast" fail-on: "critical" create-pr-comment: "true"
$ git add .github/workflows/mend-scan.yml $ git commit -m "ci: add Mend security scan workflow" [main abc1234] ci: add Mend security scan workflow 1 file changed, 28 insertions(+)
Available scan types Pass a comma-separated list to type: sca (open-source), sast (source code), container, or iac. The fail-on field accepts critical, high, medium, or low.
  • ✓ Set type: "sca,sast" to run both scanners
  • ✓ Set fail-on: "critical" to break the build on critical findings
4

Add PR Comment Integration

Configure Mend to automatically post scan results as a comment on every pull request. Reviewers can see new vulnerabilities, severity counts, and remediation links without leaving GitHub.

# Add the Mend comment-bot step to your workflow - name: Mend PR Comment if: github.event_name == 'pull_request' uses: mend-bot/comment@v1 with: mend-email: ${{ secrets.MEND_EMAIL }} mend-key: ${{ secrets.MEND_KEY }} mend-url: ${{ secrets.MEND_URL }} github-token: ${{ secrets.GITHUB_TOKEN }} show-severity: "critical,high,medium" show-fixable-only: "false"
Permissions required The permissions.pull-requests: write block from Step 3 grants the workflow the rights needed to post comments. If you omit it, the comment step will fail silently.
  • ✓ Added the mend-bot/comment step to the workflow
  • ✓ Verified the workflow has pull-requests: write permission
5

Run the Pipeline

Trigger the workflow by pushing a commit and opening a pull request. Watch GitHub Actions execute the Mend scan, then review the automated PR comment with the security summary.

# Create a feature branch and push a small change git checkout -b feature/add-config echo "LOG_LEVEL=info" >> .env.example git add .env.example git commit -m "feat: add log level config" git push origin feature/add-config # Open the pull request from the CLI gh pr create --title "Add log level config" --body "Triggers Mend scan"
$ gh pr create --title "Add log level config" --body "Triggers Mend scan" Creating pull request... https://github.com/your-org/sample-node-app/pull/42 $ gh run watch Job: mend-scan ▷ Checkout code ............... done ▷ Run Mend SCA and SAST Scan ... done ▷ Mend PR Comment ............. done Run succeeded in 1m 47s Scan results: Critical: 1 vulnerability High: 4 vulnerabilities Medium: 8 vulnerabilities PR comment posted with full report.
What to expect A new bot comment will appear on the PR titled "Mend Security Scan Results" with a table of findings, severity badges, and links to remediation guidance in the Mend platform.
  • ✓ Pushed a commit that triggered the workflow
  • ✓ Saw the Mend scan run successfully in the Actions tab
  • ✓ Verified the PR comment was posted automatically
6

Configure Quality Gates

Enforce the scan as a merge requirement. Configure branch protection so that PRs cannot be merged until the Mend scan passes, and optionally add a policy-based gate tied directly to your Mend security policy.

  • Go to Settings → Branches → Add rule for the main branch
  • Enable Require status checks to pass before merging
  • Search for and select mend-scan as a required check
  • Enable Require branches to be up to date before merging
# Optional: define a Mend policy gate that maps severity levels to pass/fail policy: gates: - name: critical-blocker severity: critical action: block - name: high-warn severity: high action: warn - name: new-findings scope: introduced severity: critical,high action: block
Heads up Once the branch protection rule is live, any new PR that introduces a critical vulnerability will be blocked from merging until the issue is remediated. Communicate this to your team before enabling it on a busy branch.
  • ✓ Added a branch protection rule on main
  • ✓ Marked mend-scan as a required status check
  • ✓ Confirmed a test PR cannot merge with a failing scan

🧠 Knowledge Check

Which on: trigger runs a workflow on both code pushes and pull requests to main?
  • on: push
  • on: [push, pull_request] with branches: [main]
  • on: workflow_dispatch
  • on: schedule
What does configuring fail-on: "critical" in the Mend action accomplish?
  • Sends an email alert when critical vulnerabilities are found
  • Fails the GitHub Actions workflow (blocking merge) if critical vulnerabilities are detected
  • Marks all vulnerabilities as critical in the exported report
  • Skips the scan when critical findings are present
