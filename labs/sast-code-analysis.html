Understanding SAST Scanning

Static Application Security Testing (SAST) analyzes source code without executing it. Mend SAST inspects the code base, tracks how untrusted data flows through the application, and identifies security flaws such as injection weaknesses, hardcoded secrets, and unsafe API usage.

Key concepts include:

  • Taint analysis marks untrusted input sources and tracks them to dangerous sinks
  • Data flow analysis follows values from entry points through functions and variables
  • Control flow analysis examines branch conditions and execution paths
# Start a SAST scan with a descriptive product name mend scan --type sast --product "Lab: Python App"
$ mend scan --type sast --product "Lab: Python App" Initializing SAST engine... Language: python Files analyzed: 42 Taint sources identified: 8 Data-flow sinks identified: 15 SAST analysis complete.
Tip SAST runs entirely on source code, so it can find issues early in the development cycle before code is deployed or executed.

In the dashboard, you'll see:

Configure Custom Rules

Extend Mend SAST with a custom rule to detect a specific pattern in your code base. Custom rules help enforce team-specific coding standards or catch risky APIs that default rules may not cover.

# Create a custom SAST rule to forbid eval() in Python mend sast rules create --name "No eval()" --pattern "eval\\(" --language python --severity high
$ mend sast rules create --name "No eval()" --pattern "eval\\(" --language python --severity high Creating custom rule... Name: No eval() Pattern: eval\( Language: python Severity: high Custom rule created successfully. Rule ID: RULE-9876
Tip Keep custom rules focused and well-documented. Overly broad patterns can produce noise and false positives, which reduce trust in scan results.
  • ✓ Created the custom "No eval()" rule
  • ✓ Verified the rule appears in the custom rules list
  • ✓ Re-ran the scan to confirm the rule is applied

Manage False Positives

Not every finding is exploitable. Triage findings to reduce noise, mark false positives with a reason, and establish a baseline so future scans only report new issues.

# Mark a specific finding as a false positive mend sast triage --finding F-1234 --status false-positive --reason "Sanitized input" # Create a baseline from the current scan mend sast baseline --create --output baseline.json
$ mend sast triage --finding F-1234 --status false-positive --reason "Sanitized input" Triaging finding F-1234... Status: false-positive Reason: Sanitized input Finding F-1234 triaged successfully. $ mend sast baseline --create --output baseline.json Creating baseline... Total findings: 7 Triaged findings: 1 Baseline written to baseline.json
Warning Only mark findings as false positives when you are confident the data is sanitized or the sink is unreachable. Document the reason so the decision can be audited later.
  • ✓ Triaged finding F-1234 with a clear reason
  • ✓ Created a baseline file
  • ✓ Confirmed the false positive is excluded from the active count

Apply AI Remediation

Mend AI can suggest secure replacements for vulnerable code. Review the proposed fix, validate it in context, and apply it to the source file.

# Request AI remediation for a specific finding mend sast remediate --finding F-5678 --apply
$ mend sast remediate --finding F-5678 --apply Generating AI remediation for finding F-5678... Vulnerability: SQL injection in routes.py:42 Suggested fix: cursor.execute("SELECT * FROM products WHERE name = ?", (query,)) Applying fix... Updated routes.py Re-run the scan to verify the finding is resolved.
Before you apply Always review AI suggestions in the context of your application. Test the fix locally and confirm that behavior remains correct before committing the change.
